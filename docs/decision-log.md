## ADR-001：采用 Spring Boot 单体架构

- **状态：** Accepted
- **日期：** 2026-07-12
- **背景：** 项目是学习型 MVP，服务器配置为 2 核 2G，业务量很小。
- **候选方案：**
  1. Spring Boot 单体
  2. Spring Cloud 微服务
- **决策：** 采用 Spring Boot 单体 + 模块化分包。
- **原因：**
  - 功能规模小，没有独立部署和扩缩容需求；
  - 微服务增加服务注册、网关、配置中心、链路追踪等运维复杂度；
  - 当前学习重点应放在业务闭环、权限、事务和订单状态上。
- **后果：** 开发和部署简单，适合单机服务器，调试和理解容易。后续业务增长时需要重新评估拆分。

## ADR-002：采用数据库唯一索引作为并发预约最终保障

- **状态：** Accepted
- **日期：** 2026-07-12
- **背景：** 同一阿姨同一小时块只能被一位阿姨抢到，需要解决并发冲突。
- **候选方案：**
  1. 数据库唯一索引 + 事务
  2. Redis 分布式锁
  3. 先查再插入（无锁方案）
- **决策：** 采用方案 1：数据库唯一索引 + 事务 + 订单状态条件更新。
- **原因：**
  - 唯一索引是数据库层面的原子性保障，可靠且容易理解；
  - Redis 分布式锁增加复杂度和运维成本，第一版业务量不需要；
  - "先查再插入"在并发下存在竞态条件，不可靠。
- **后果：** 需要在代码中捕获 `DuplicateKeyException` 并转换为友好的业务提示。

## ADR-003：第一版采用 9 状态订单状态机

- **状态：** Accepted
- **日期：** 2026-07-12
- **背景：** 需要支持阿姨抢单、阿姨完成服务、用户确认、投诉等完整流程。
- **候选方案：**
  1. 5 状态：待支付 → 待服务 → 待评价 → 已完成 → 已取消
  2. 9 状态：待支付 → 待抢单 → 待服务 → 服务中 → 待确认 → 待评价 → 已完成 → 已取消 → 投诉中
- **决策：** 采用 9 状态。
- **原因：** 5 状态无法区分阿姨抢单、服务中、用户确认等环节，不利于状态追踪和权限控制。
- **后果：** 状态流转逻辑更复杂，但业务表达更精确，每个角色的操作边界更清晰。

## ADR-004：阿姨端与用户端共用同一个 Vue 前端工程

- **状态：** Accepted
- **日期：** 2026-07-12
- **背景：** 需要支持用户和阿姨两种角色，但第一版不做独立 App。
- **候选方案：**
  1. 三个独立前端工程（用户端 / 阿姨端 / 管理后台）
  2. 两个前端工程（用户阿姨共用 + 管理后台）
  3. 一个前端工程（三个角色路由分发）
- **决策：** 采用方案 3。
- **原因：** 前端代码量不大，三个角色共用登录、订单列表等组件，减少重复开发。
- **后果：** 前端打包体积会包含所有角色的代码，但第一版可以接受。后续可根据复杂度拆分。

## ADR-005：第一版不使用超时自动取消

- **状态：** Accepted
- **日期：** 2026-07-12
- **背景：** 原计划待支付订单 30 分钟自动取消。
- **候选方案：**
  1. Spring Task 每分钟扫描 + 自动取消
  2. 不设自动取消，仅保留用户主动取消
- **决策：** 采用方案 2。
- **原因：** 第一版是学习型项目，没有大量真实订单，自动取消功能的学习价值低于其他核心功能。
- **后果：** 待抢单订单可能长期存在无人处理的状态，但管理员可以手动介入。

## ADR-006：第一版不做独立地址表

- **状态：** Accepted
- **日期：** 2026-07-12
- **背景：** 原计划有地址管理功能（CRUD + 默认地址）。
- **候选方案：**
  1. 独立 address 表，支持 CRUD
  2. 地址直接作为订单字段保存，不做地址管理
- **决策：** 采用方案 2。
- **原因：** 地址 CRUD 的复杂度不高但有重复的增删改查工作量，第一版订单流程更需要优先实现。
- **后果：** 后续可以通过表结构变更将地址字段抽取为独立表，升级为完整地址管理。

## ADR-007：采用整点小时块替代固定时段

- **状态：** Accepted
- **日期：** 2026-07-12
- **背景：** 原计划固定上午（09-12）和下午（14-17）两个时段。
- **候选方案：**
  1. 固定两个 3 小时时段
  2. 用户自由选择开始时间 + 时长（整点小时块）
- **决策：** 采用方案 2。
- **原因：** 小时块更灵活，能覆盖更多服务和演示场景；档期控制逻辑也更通用，未来扩展时不需要重构。
- **后果：** 档期表设计需要按小时块存储，比固定时段更复杂，但灵活性更高。

## ADR-008：阿姨状态分离为管理与接单两个字段

- **状态：** Accepted
- **日期：** 2026-07-12
- **背景：** 阿姨需要同时支持管理员控制（上下架/禁用）和阿姨自主控制（休息/接单），以及系统控制的档期占用。
- **候选方案：**
  1. 一个字段混合表达所有状态
  2. 两个独立字段：管理状态 + 接单状态，档期由独立表管理
- **决策：** 采用方案 2。
- **原因：** 一个字段无法区分"管理员下架"和"阿姨休息"——前者用户不可见，后者只是阿姨自主设置。混合表达会导致逻辑混乱和权限漏洞。
- **后果：** 需要两个字段 + 一个档期表，模型更清晰但字段更多。

## ADR-009：阿姨自行注册，管理员只负责运营状态管理

- **状态：** Accepted
- **日期：** 2026-07-12
- **决策：** 阿姨通过公开注册接口自行创建账号和一对一资料，管理员不能新增或代注册阿姨，只能编辑资料、上下架、禁用和逻辑删除。
- **安全约束：** 公开注册角色只允许 USER/AUNT；ADMIN 仅由初始化 SQL 创建；角色由后端白名单校验。
- **后果：** 注册事务需要同时创建 `sys_user` 与 `aunt`；存在历史订单的阿姨禁止物理删除。

## ADR-010：拆分订单归属并发与阿姨档期并发保障

- **状态：** Accepted
- **日期：** 2026-07-12
- **决策：** 同一订单只能被抢一次由订单状态条件更新保障；同一阿姨同一小时块冲突由联合唯一索引保障，两者在同一事务内执行。
- **后果：** 必须校验条件更新行数，档期插入失败时回滚订单归属。

- **状态：** Accepted

## ADR-013：ADMIN 账号由应用启动初始化，密码走环境变量

- **状态：** Accepted
- **日期：** 2026-07-13
- **背景：** 阶段 1 需要创建管理员账号。规范要求"管理员账号仅由初始化 SQL 创建"，但 ADR-012 要求"管理员初始密码不得写入源码、日志或 Git"。
- **候选方案：**
  1. init.sql 写明文密码插入 ADMIN
  2. init.sql 写固定 BCrypt 哈希插入 ADMIN（对应固定默认密码）
  3. 应用启动时由 ApplicationRunner 用环境变量密码 BCrypt 加密后插入（init.sql 只建表）
- **决策：** 采用方案 3。
- **原因：**
  - 方案 1 明文密码进 SQL 文件入 Git，直接违反 ADR-012；
  - 方案 2 固定哈希入 Git，虽不可逆但可离线爆破，且默认密码固定存在安全风险；
  - 方案 3 密码只走环境变量 `ADMIN_INIT_PASSWORD`，BCrypt 哈希入库，符合 ADR-012「敏感配置仅环境变量注入」；建表仍由 SQL（schema.sql）完成，ADMIN 创建是系统初始化逻辑而非注册接口，不违反"管理员不开放注册"。
- **后果：**
  - 部署时必须设置 `ADMIN_INIT_PASSWORD` 环境变量，否则启动跳过 ADMIN 初始化并日志告警；
  - 幂等：已存在 ADMIN 账号则跳过，可重复启动；
  - 默认手机号 `13800000000`，可由 `ADMIN_PHONE` 环境变量覆盖。

## ADR-014：JWT 密码哈希签名使改密码后旧 token 失效

- **状态：** Accepted
- **日期：** 2026-07-13
- **背景：** 无状态 JWT 改密码后旧 token 仍有效直到过期，存在安全风险（token 泄露后改密码无法立即止损）。
- **候选方案：**
  1. Redis JWT 黑名单（改密码时把旧 token 加入黑名单）
  2. token_version 版本号字段（sys_user 加字段，改密码时 +1，校验时比对）
  3. 密码哈希签名（JWT claims 存密码哈希前 16 位，Filter 校验时比对数据库当前哈希）
- **决策：** 采用方案 3。
- **原因：**
  - 方案 1 不适用：找回密码接口（手机号 + 验证码）不携带旧 token，无法获取要拉黑的 token；
  - 方案 2 需给 sys_user 加字段 + 改 resetPassword 逻辑，改动较大；
  - 方案 3 无需改表结构，改密码后 BCrypt 重新生成 salt，哈希自然变化，旧 token 的 pwdSig 自动失效，改动最小。
- **后果：**
  - 每次受保护请求查一次 sys_user 比对 pwdSig（MVP 可接受，后续可 Redis 缓存降低 DB 压力）；
  - JWT claims 含密码哈希前 16 位（BCrypt 哈希片段，含 salt 前缀，非明文密码，安全性可接受）；
  - 改密码后旧 token 立即失效，新登录获取的新 token 有效。

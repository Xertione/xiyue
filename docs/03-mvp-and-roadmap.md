# 第一部分：第一版 MVP

## 1. MVP 定位

息悦生活是一个面向"日常保洁预约"场景的前后端分离平台。

第一版定位为：

- 学习型、简历型项目；
- 不进行正式商业化运营；
- 采用一个 Vue 3 前端工程，用户/阿姨/管理员根据角色进入不同路由；
- 不做小程序、独立 App、真实支付、真实短信；
- 重点实现认证授权、角色路由、阿姨抢单、订单状态机、小时块档期冲突控制、事务一致性、部署流程。

## 2. 用户角色

### 2.1 普通用户

普通用户可以：

- 手机号 + 验证码登录，或手机号 + 密码登录；
- 注册账号（手机号 + 密码 + 固定验证码）；
- 找回密码（手机号 + 固定验证码 + 新密码）；
- 浏览可接单阿姨；
- 按星级、价格、技能标签筛选阿姨；
- 查看阿姨详情；
- 发布日常保洁订单（选择日期、开始时间、服务时长，填写联系人姓名、电话、地址）；
- 对待支付订单进行模拟支付；
- 对待支付订单进行取消；
- 查看自己的订单列表和详情；
- 确认阿姨提交的服务完成（服务履约最后一步）；
- 在服务确认完成后提交一次评价；
- 对待评价订单提交一次投诉（文字原因，订单进入投诉中）。

### 2.2 阿姨

阿姨可以登录系统并自主抢单。阿姨与用户共用同一套 Vue 前端工程，登录后根据角色进入阿姨路由。

阿姨可以做：

- 手机号 + 验证码登录，或手机号 + 密码登录；
- 自行注册阿姨账号（手机号 + 密码 + 固定验证码，角色由后端白名单校验）；
- 浏览抢单大厅中的待抢单订单；
- 抢单（一个订单只能被一位阿姨抢到）；
- 查看自己的订单；
- 开始服务（将待服务订单变为服务中）；
- 提交服务完成和演示图片 URL（将服务中订单变为待确认）；
- 设置个人休息/可抢单状态；
- 设置具体日期和小时块不可接单。

第一版不提供：

- 阿姨收入、提现和结算；
- 阿姨培训；
- 实时定位；
- 聊天；
- 复杂排班。

### 2.3 管理员

管理员可以：

- 使用预置账号密码登录管理后台；
- 查看和编辑已自行注册的阿姨资料；
- 上架、下架、禁用或逻辑删除阿姨；已有历史订单的阿姨禁止物理删除；
- 查看全量订单；
- 按状态筛选订单；
- 查看订单详情；
- 在后台手动指派阿姨到待抢单订单（作为兜底）；
- 查看评价和投诉；
- 手动处理投诉。

## 3. 服务规则

### 3.1 服务品类

第一版只提供一个服务品类：

```text
日常保洁
```

### 3.2 服务时间规则

采用整点小时块方式，不支持用户输入任意分钟时间范围：

```text
- 用户选择服务日期；
- 用户选择开始时间（整点，如 09:00、10:00、14:00）；
- 用户选择服务时长（按整小时计算，最低 1 小时）；
- 系统内部按小时块锁定阿姨档期。

示例：
日期：2026-08-01
开始：10:00
时长：3 小时

占用小时块：
10:00 - 11:00
11:00 - 12:00
12:00 - 13:00
```

### 3.3 价格规则

两种价格规则并存：

```text
1. 抢单订单：
   - 用户发布订单时，订单本身已确定单价和总金额；
   - 阿姨抢到订单后，按订单金额结算；
   - 即使阿姨个人标价更高，抢低价订单时也按订单价格结算。

2. 指定阿姨订单（辅助模式）：
   - 用户指定阿姨时，订单按该阿姨当前标价计算；
   - 用户需按指定阿姨价格支付。
```

第一版订单必须保存价格快照，避免阿姨后续修改标价影响历史订单。

### 3.4 预约与抢单规则

```text
1. 用户可以浏览可接单阿姨列表和抢单大厅；
2. 用户只能预约未来日期；
3. 系统不限制阿姨可接单的订单数量，阿姨根据自己档期决定是否抢单；
4. 阿姨抢单成功后，锁定该阿姨对应的日期和小时块；
5. 待支付订单和待抢单订单均不占用任何阿姨档期；
6. 一个订单只能被一位阿姨抢到（通过事务+唯一索引保障）；
7. 订单取消后释放对应的档期占用记录；
8. 第一版不支持改期；
9. 第一版不支持多个阿姨共同服务一个订单；
10. 第一版不实现复杂智能派单算法。
```

## 4. 订单状态机

### 4.1 状态定义

| 状态值 | 状态名称 | 含义 |
|---:|---|---|
| 0 | 待支付 | 用户创建订单成功，尚未支付 |
| 1 | 待抢单 | 用户模拟支付成功，等待阿姨抢单或管理员指派 |
| 2 | 待服务 | 阿姨已抢单或被指派，等待阿姨开始服务 |
| 3 | 服务中 | 阿姨开始服务 |
| 4 | 待确认 | 阿姨提交服务完成，等待用户确认 |
| 5 | 待评价 | 用户确认服务完成，等待用户评价 |
| 6 | 已完成 | 用户已提交评价，订单闭环 |
| 7 | 已取消 | 用户主动取消订单 |
| 8 | 投诉中 | 用户提交投诉，管理员处理中 |

### 4.2 合法状态流转

```text
用户创建订单
└── 待支付
    ├── 模拟支付成功 ──────────> 待抢单
    ├── 用户取消 ──────────────> 已取消
    └── [不设超时自动取消]

待抢单
    ├── 阿姨抢单 / 管理员指派 ──> 待服务
    ├── 用户取消 ──────────────> 已取消
    └── [暂不处理长时间无人抢单]

待服务
    ├── 阿姨开始服务 ──────────> 服务中
    └── 用户取消 ──────────────> 已取消

服务中
    ├── 阿姨提交服务完成 ──────> 待确认
    └── [禁止直接取消，MVP 由管理员线下介入]

待确认
    ├── 用户确认服务完成 ──────> 待评价
    ├── [禁止直接取消，MVP 由管理员线下介入]
    └── [暂不处理用户不确认]

待评价
    ├── 用户提交评价 ──────────> 已完成
    └── 用户提交投诉 ──────────> 投诉中

投诉中
    └── 管理员手动处理 ────────> 已完成
```

### 4.3 状态约束

```text
- 只有待支付订单可以执行模拟支付；
- 只有待支付、待抢单、待服务订单可以取消；服务中和待确认订单禁止直接取消；
- 只有待抢单订单可以执行阿姨抢单操作；
- 抢单大厅价格以订单价格快照为准；阿姨抢单即表示接受该价格，不与个人标价比较；
- 只有待服务订单可以开始服务；
- 只有服务中订单可以提交服务完成；
- 只有待确认订单可以确认服务完成；
- 只有待评价订单可以评价或投诉；
- 同一订单仅允许评价一次；
- 所有状态变更均需校验订单当前状态；
- 所有状态更新应使用带条件的 UPDATE 语句，避免并发下重复处理。
```

## 5. MVP 数据模型

### 5.1 核心表

| 表名 | 作用 |
|---|---|
| `sys_user` | 用户、阿姨、管理员账号（由角色字段区分） |
| `aunt` | 阿姨资料（独立于 sys_user 的阿姨个人信息和运营信息） |
| `service_order` | 预约订单 |
| `aunt_booking_slot` | 阿姨小时块档期占用记录 |
| `review` | 用户评价 |
| `complaint` | 用户投诉（第一版简化实现） |

第一版不做独立地址表，地址信息直接作为订单字段保存。

### 5.2 关键设计原则

#### 用户表 `sys_user`

- `phone` 必须唯一；
- 使用角色字段区分普通用户（USER）、阿姨（AUNT）、管理员（ADMIN）；
- 所有角色均使用 BCrypt 密码存储；
- 支持三种认证方式：密码登录、固定验证码登录、验证码+密码找回密码；
- 注册时写入手机号、密码（BCrypt）和角色；角色只允许 `USER` 或 `AUNT`，禁止注册 `ADMIN`；
- 注册角色必须由后端白名单校验，不能直接信任任意客户端角色值。

#### 阿姨表 `aunt`

包含：

```text
姓名、头像、标价（用于指定阿姨模式）、星级、服务次数、技能标签、介绍、
管理员状态（可用/下架/禁用）、接单状态（可抢单/休息）、创建时间
```

注意：管理员状态和阿姨自己的接单状态使用不同字段，不允许混合。
`aunt.user_id` 必须唯一并关联自行注册的 `sys_user`。删除阿姨默认使用逻辑删除；存在历史订单时禁止物理删除。

#### 地址

第一版不做地址管理。联系人、联系电话、服务详细地址直接作为 `service_order` 字段保存。

#### 订单表 `service_order`

订单必须保存下单时的快照，至少包括：

```text
订单号（唯一）
用户 ID
阿姨 ID（抢单/指派后写入）

阿姨姓名快照
阿姨头像快照

服务名称快照
服务日期
开始时间（整点）
服务时长（小时数）

联系人姓名（订单快照）
联系人电话（订单快照）
服务详细地址（订单快照）

订单金额
阿姨标价（指定阿姨模式下用）
价格类型（抢单价 / 标价）
支付方式
模拟支付流水号
模拟退款状态
模拟退款流水号
模拟退款时间
订单状态

支付时间
服务完成提交时间
用户确认时间
取消时间
取消原因
创建时间
更新时间
```

#### 档期占用表 `aunt_booking_slot`

按小时块设计：

```text
id（主键）
aunt_id
service_date
hour_slot（如 9、10、14——整点小时数）
order_id
create_time
```

唯一索引：

```sql
UNIQUE KEY uk_aunt_date_hour (aunt_id, service_date, hour_slot)
```

档期在阿姨抢单成功或管理员指派阿姨成功时才锁定。待支付/待抢单订单不锁定档期。

#### 评价表 `review`

包含：

```text
订单 ID、用户 ID、阿姨 ID、评分（1-5）、评价内容、创建时间
```

唯一索引：

```sql
UNIQUE KEY uk_review_order (order_id)
```

第一版只做评分+文字评价，不做追评、图片评价、标签评价。

#### 投诉表 `complaint`

第一版简化实现：

```text
订单 ID、用户 ID、投诉原因文字、投诉说明文字、管理员处理备注、状态（待处理/已处理）、创建时间、处理时间
```

`complaint.order_id` 必须唯一。同一订单只能投诉一次；投诉处理完成后订单进入已完成且不再允许评价。

## 8. MVP 中必须落实的技术点

### 8.1 认证与注册

- 手机号 + BCrypt 密码注册，公开角色仅允许 `USER` 或 `AUNT`；
- 手机号 + 密码登录；
- 手机号 + 固定验证码 `123456` 登录（验证码缓存在 Redis）；
- 手机号 + 固定验证码 + 新密码找回密码；
- JWT 签发与解析；
- 所有角色共用 JWT，角色信息编码在 Token 中；
- 管理员账号仅由初始化 SQL 创建，不开放注册；
- JWT 角色来源必须是数据库账号信息，不能直接信任客户端提交的角色。

### 8.2 权限控制

- 使用 Spring Security + JWT Filter；
- 用户/阿姨/管理员三种角色接口隔离；
- 用户只能访问自己的订单、评价、投诉；
- 阿姨只能查看和操作自己接的订单；
- 管理员可以查看全量数据；
- 所有涉及 ID 的接口都要检查资源归属，防止水平越权。

### 8.3 阿姨抢单事务一致性

阿姨抢单或管理员指派阿姨时，必须在同一个事务内完成：

```text
校验订单当前状态为待抢单
→ 校验阿姨管理状态为可用
→ 校验阿姨的该日期和小时块未被占用
→ 插入阿姨小时块档期占用记录
→ 更新订单 aunt_id
→ 更新订单状态为待服务
```

任一步失败都必须回滚。程序需要捕获唯一索引冲突并返回明确业务提示。

### 8.4 用户创建订单事务

用户创建订单时，在同一个事务内完成：

```text
校验用户注册状态
→ 保存订单（含地址快照、价格快照）
→ 订单状态初始化为待支付
```

注意：创建订单时不插入档期占用记录，也不锁定阿姨。

### 8.5 并发抢单控制

```text
多个阿姨同时抢同一订单时，必须依靠订单状态条件更新确保只有一个请求成功：
`UPDATE service_order ... WHERE id = ? AND status = '待抢单' AND aunt_id IS NULL`。
更新行数必须为 1，否则返回“订单已被抢”。
档期联合唯一索引只负责防止同一阿姨同一小时块重复占用，不能代替订单条件更新。
订单条件更新与全部小时块插入必须位于同一事务，任一步失败都整体回滚。
```

### 8.6 超时/取消处理

第一版不设置强制自动取消规则：

- 用户仅可取消待支付、待抢单、待服务订单；
- 服务中和待确认订单禁止直接取消，MVP 由管理员线下介入；系统投诉仅在待评价状态开放；
- 已支付订单取消时，记录模拟退款状态、流水号和时间；
- 已有阿姨接单的订单取消时，同步释放对应小时块档期；
- 取消必须使用状态条件更新，避免与抢单或开始服务并发成功；
- 暂不处理长时间无人抢单的自动取消。

### 8.7 订单快照

订单创建时必须保存地址、阿姨和价格信息快照。阿姨在抢单成功时将本人的姓名、头像写入订单快照。

### 8.8 阿姨状态分离

阿姨存在两类独立的状态字段：

```text
- 管理状态（管理员控制）：可用 / 下架 / 禁用
- 接单状态（阿姨或系统控制）：可抢单 / 休息 / 具体小时块占用
```

禁止用一个字段混合表达以上不同状态。

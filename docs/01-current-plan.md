## 3. 初始 MVP 功能设想

### 3.1 用户端功能

- 手机号 + 模拟验证码登录；
- 浏览阿姨列表；
- 按星级、价格筛选阿姨；
- 查看阿姨详情；
- 选择服务日期与固定时段；
- 填写或选择服务地址；
- 创建预约订单；
- 模拟支付；
- 查看订单列表与订单详情；
- 服务完成后提交评分和文字评价。

### 3.2 管理后台功能

- 管理员登录；
- 阿姨新增、编辑、上下架；
- 查看全量订单；
- 按订单状态筛选；
- 管理员标记订单服务完成；
- 查看用户评价。

### 3.3 初始订单流程设想

```text
用户登录
→ 浏览阿姨
→ 选择阿姨、日期、服务时段和服务地址
→ 创建待支付订单
→ 模拟支付
→ 订单进入待服务
→ 管理员标记服务完成
→ 用户评价
→ 订单完成
```

### 3.4 初始订单状态设想

```text
待支付
  ├─ 用户模拟支付 → 待服务
  └─ 超时未支付 / 用户取消 → 已取消

待服务
  └─ 管理员标记服务完成 → 待评价

待评价
  └─ 用户评价 → 已完成
```

## 4. 初始数据表设想

初始规划涉及以下核心表：

| 表名（初始） | 用途 |
|---|---|
| `user` | 普通用户和管理员账号 |
| `aunt` | 阿姨基础资料 |
| `order` | 订单信息 |
| `review` | 用户评价 |
| `address` | 用户常用地址 |
| `aunt_schedule` | 阿姨档期或预约占用信息 |

> 注：最终表名和字段设计需要避免 SQL 关键字，例如不建议直接使用 `order` 作为表名，后续预计调整为 `service_order`。

## 5. 初始接口设想

| 接口 | 方法 | 初始用途 |
|---|---|---|
| `/api/user/login` | POST | 手机号 + 验证码模拟登录 |
| `/api/aunt/list` | GET | 阿姨列表 |
| `/api/aunt/{id}` | GET | 阿姨详情 |
| `/api/order/create` | POST | 创建订单 |
| `/api/order/pay` | POST | 模拟支付 |
| `/api/order/list` | GET | 用户订单列表 |
| `/api/review/create` | POST | 提交评价 |

> 注：接口路径、资源命名、权限设计将在最终方案中统一调整。

## 8. 当前项目最重要的真实目标

本项目的第一目标不是堆叠技术名词，而是：

1. 独立完成一个完整的前后端分离项目；
2. 理解 Spring Boot、MyBatis-Plus、Spring Security、JWT、MySQL、Redis、Docker Compose 的基本使用方式；
3. 理解订单状态流转、权限校验、事务、并发预约等真实业务问题；
4. 能够自己部署、排错、测试和维护项目；
5. 最终能够在面试中解释项目的设计、问题、取舍和解决方案。

# 息悦生活开发时间计划

> 项目目标：完成可部署、可演示、可解释的 MVP。  
> 推荐周期：4～6 周。  
> 原则：每个阶段完成后必须进行测试、Git 提交、文档记录，不盲目赶进度。  
> 本次更新：2026-07-12，对齐已确认的 9 状态订单状态机、小时块档期、阿姨抢单模式。

---

## 1. 总体时间安排

| 阶段 | 建议时长 | 核心产出 |
|---:|---|---|
| 阶段 0：项目基础设施 | 2 - 3 天 | Docker 环境、Spring Boot 骨架、统一响应体、API 文档 |
| 阶段 1：认证与注册 | 3 - 5 天 | 注册/登录/找回密码、JWT、Spring Security、Redis 验证码 |
| 阶段 2：阿姨管理 | 3 - 5 天 | 阿姨自行注册资料、管理员状态管理、用户端列表详情、筛选 |
| 阶段 3：订单与抢单 | 5 - 8 天 | 创建订单、模拟支付、抢单大厅、阿姨抢单、事务+唯一索引 |
| 阶段 4：服务履约与评价 | 4 - 6 天 | 阿姨开始服务、提交完成、用户确认、评价、投诉 |
| 阶段 5：联调、部署、复盘 | 5 - 8 天 | 前端页面、Docker、Nginx、云服务器、演示 |
| 机动时间 | 3 - 5 天 | 排错、重构、补测试、补文档 |

---

## 2. 第一周：基础环境、骨架、认证

### Day 1：Docker 基础环境

目标：

- 确认 Docker、Docker Compose 已安装；
- 编写 Docker Compose 配置（MySQL 8 + Redis 7，含数据卷、日志滚动、资源限制）；
- 启动容器，使用数据库客户端连接 MySQL；
- 建立数据库 `xiyue_life`。

验收：

```bash
docker compose ps
docker compose logs mysql
docker compose logs redis
```

---

### Day 2：Spring Boot 项目骨架

目标：

- 创建 Spring Boot 3 + JDK 17 + Maven 项目；
- 配置 MySQL 数据源；
- 配置 Redis（固定验证码缓存）；
- 集成 MyBatis-Plus；
- 建立统一返回体；
- 建立全局异常处理；
- 创建健康检查接口；
- 集成 Springdoc OpenAPI + Knife4j。

验收：

```text
GET /api/health 返回成功；
项目可连接 MySQL；
项目启动无报错；
API 文档可访问。
```

---

### Day 3 - Day 4：认证与注册

目标：

- 创建 `sys_user` 表（含 USER / AUNT / ADMIN 三种角色）；
- 数据库初始化 SQL 预置管理员账号；
- 注册接口（手机号 + BCrypt 密码 + 固定验证码 + USER/AUNT 白名单角色）；
- 禁止公开注册 ADMIN，管理员账号仅由初始化 SQL 创建；
- 密码登录接口；
- 固定验证码登录接口（验证码 Key：`sms:code:{phone}`，TTL 5 分钟）；
- 找回密码接口（验证码 + 新密码）；
- JWT 签发与解析；
- Spring Security + JWT Filter 配置。

验收：

```text
- 用户可注册；
- 未携带 Token 访问受保护接口：401；
- 不同角色访问不属于自己的接口：403；
- 登录后可获取用户身份。
```

---

### Day 5：前端基础与登录页

目标：

- 初始化 Vue 3 + Vite + TypeScript 前端工程；
- 配置 Pinia（登录状态、用户信息、角色信息）；
- 配置 Axios 请求拦截器（自动附加 Token）；
- 编写用户/阿姨公用登录页和注册页；
- 编写管理后台登录页；
- 前端保存 Token，处理 Token 失效。

---

## 3. 第二周：阿姨管理与用户端

### Day 6 - Day 7：阿姨管理后端

目标：

- 创建 `aunt` 表；
- 阿姨两类状态字段设计（管理状态 + 接单状态，不可混合）；
- 阿姨自行注册时初始化一对一 `aunt` 资料；
- 管理员编辑、上下架、禁用或逻辑删除阿姨；
- 阿姨列表分页查询；
- 按星级、价格、技能标签筛选；
- 阿姨详情接口；
- 阿姨设置个人接单状态（可抢单/休息）；
- 初始化测试阿姨数据（含不同价格、星级、标签）。

验收：

```text
阿姨可自行注册，管理员不能新增阿姨；
下架阿姨不在用户端列表出现；
用户端只能看到可接单状态的阿姨。
```

---

### Day 8 - Day 9：阿姨管理与用户端页面

目标：

- 管理后台阿姨列表、编辑、状态管理和逻辑删除页面；
- 用户端阿姨列表页（含筛选）；
- 用户端阿姨详情页；
- 阿姨端的接单状态设置页面。

---

## 4. 第三周：订单与抢单

### Day 10 - Day 11：订单表和档期表

目标：

- 创建 `service_order` 表（含地址快照、价格快照，9 个订单状态枚举）；
- 创建 `aunt_booking_slot` 表（按小时块）；
- 添加唯一索引 `uk_aunt_date_hour`；
- 设计订单状态枚举（9 状态）；
- 设计订单号生成规则；
- 设计创建订单 DTO（日期、开始时间、时长、联系人、电话、地址）。

---

### Day 12 - Day 14：创建订单与模拟支付

目标：

- 用户创建订单（含地址快照、价格快照），状态初始化为待支付；
- 注意：创建订单时不锁定任何阿姨档期；
- 模拟支付（MockPaymentService），生成流水号，记录支付时间；
- 支付成功后订单状态变为待抢单；
- 用户取消待支付订单。

验收：

```text
用户可以创建待支付订单；
订单保存了地址和价格快照；
支付后订单进入待抢单。
```

---

### Day 15：抢单大厅

目标：

- 前端抢单大厅页面（显示所有待抢单订单）；
- 阿姨查看待抢单订单列表和详情；
- 阿姨抢单（事务：校验状态和档期 → 条件更新订单归属 → 插入全部档期；任一步失败整体回滚）；
- 捕获唯一索引冲突，返回明确业务提示；
- 管理员后台手动指派阿姨（兜底）。

验收：

```text
阿姨可看到所有待抢单订单；
抢单成功后订单变为待服务；
重复抢同一订单失败并提示。
```

---

### Day 16：并发测试与取消

目标：

- 使用 Apifox 并发调用抢单接口；
- 多个阿姨同时抢同一订单 → 仅一人成功；
- 同一阿姨抢时间冲突订单 → 仅一次成功；
- 用户取消待支付、待抢单、待服务订单（含模拟退款字段 + 档期释放）；
- 验证服务中和待确认订单不能直接取消。

本阶段必须记录到 troubleshooting / interview-notes：

```text
为什么不用"先查再插入"解决并发；
为什么数据库唯一索引是最终保障；
事务和条件更新分别解决什么问题。
```

---

## 5. 第四周：服务履约、评价、投诉

### Day 17 - Day 18：阿姨服务流程

目标：

- 阿姨开始服务（待服务 → 服务中）；
- 阿姨提交服务完成（服务中 → 待确认，上传演示图片 URL）；
- 用户确认服务完成（待确认 → 待评价）；
- 服务中和待确认订单禁止直接取消，MVP 由管理员线下介入；进入待评价后才开放系统投诉。

验收：

```text
阿姨可以开始和完成服务；
用户可以看到阿姨提交的完成状态；
用户确认后订单进入待评价。
```

---

### Day 19：评价功能

目标：

- 创建 `review` 表（含唯一索引）；
- 用户对待评价订单提交评分（1-5）和文字评价；
- 防止重复评价；
- 评价后订单变为已完成；
- 更新阿姨服务次数和星级。

---

### Day 20：投诉功能

目标：

- 创建 `complaint` 表并为 `order_id` 添加唯一索引；
- 仅待评价订单允许提交一次投诉，订单变投诉中；
- 管理员后台查看投诉内容；
- 管理员手动处理投诉（投诉中 → 已完成）。

---

### Day 21：第四周复盘

复盘重点：

```text
订单状态机（9 状态）；
为什么状态更新要带条件（WHERE status=xx）；
模拟支付与真实支付的差别；
评价为什么要唯一约束；
订单快照为什么重要；
阿姨状态分离的设计思路。
```

## 6. 第五周：前端完善、部署、演示

### Day 22 - Day 24：全流程联调

目标：

- 用户端完整页面（注册、登录、阿姨列表、发布订单、订单列表、评价、投诉）；
- 阿姨端完整页面（登录、抢单大厅、我的订单、开始服务、提交完成）；
- 管理后台完整页面（登录、阿姨管理、订单管理、投诉处理）；
- 优化错误提示、空状态、加载状态；
- 使用测试账号跑完整演示流程。

完整演示流程：

```text
用户注册/登录
→ 浏览阿姨
→ 发布订单（选择日期、小时、填写地址）
→ 模拟支付
→ 订单进入待抢单
→ 阿姨登录
→ 阿姨抢单
→ 阿姨开始服务
→ 阿姨提交服务完成
→ 用户确认服务完成
→ 用户评价 / 投诉
→ 订单完成
```

# 我如何使用 AI 开发息悦生活项目

> 这是一份写给自己的项目协作原则。  
> 目标不是让 AI 替我完成项目，而是利用 AI 加快理解、排错、验证和复盘。

---

## 1. 我做这个项目的真实目标

我做息悦生活，不只是为了得到一个能运行的项目。

我更希望通过它学习并掌握：

- Spring Boot 项目如何组织；
- 请求如何从前端到 Controller、Service、Mapper、MySQL；
- JWT 如何识别登录用户；
- Spring Security 如何控制接口权限；
- MyBatis-Plus 如何操作数据库；
- Redis 在真实项目中适合做什么；
- 数据库事务为什么重要；
- 并发下单为什么会重复；
- 唯一索引、条件更新如何解决业务冲突；
- Docker Compose 如何部署多个服务；
- Nginx 为什么可以代理前端和后端；
- 遇到报错时如何自己定位问题。

项目的成功标准不是"AI 帮我写完了多少代码"，而是：

```text
我能解释每个模块为什么存在；
我能说清一个请求的完整链路；
我知道代码出错时先看哪里；
我能说明自己遇到过什么问题、怎么定位、怎么解决；
我能在面试中真实地讲出项目的设计与取舍。
```

---

## 2. 我不应该怎样使用 AI

以下方式要尽量避免：

### 2.1 不要一次性让 AI 生成整个项目

不要这样问：

```text
帮我生成一个完整的家政平台，
要有 Spring Boot、JWT、Redis、Vue、订单、支付、Docker、后台……
```

问题是：

- 代码量巨大；
- 不知道从哪里开始看；
- 出错后无法定位；
- 需求一变就不知道怎么改；
- 最后只是在复制粘贴；
- 面试时无法解释。

---

### 2.2 不要只问"报错怎么办"

不要只发一句：

```text
报错了，怎么办？
```

应该尽量提供：

```text
1. 我想实现什么；
2. 实际发生了什么；
3. 完整报错信息；
4. 相关代码；
5. 我已经尝试过什么；
6. 我猜测问题在哪里。
```

例如：

```text
我在 Spring Security 6 中配置了 JwtAuthenticationFilter，
希望访问 /api/orders 时能从请求头解析 JWT。

实际情况：请求能进入 Controller，但 SecurityContext 中没有当前用户。

以下是：
1. SecurityConfig；
2. JwtAuthenticationFilter；
3. 请求头；
4. 控制台日志。

请先分析过滤器是否正确注册到过滤器链，
不要直接重写整个认证模块。
```

这种问法会让我更快理解问题，也让 AI 更容易给出准确答案。

---

## 3. 每次只完成一个小目标

正确节奏不是：

```text
"今天完成订单模块"
```

而是：

```text
"今天完成创建订单接口的参数校验"
"今天完成地址归属校验"
"今天完成档期唯一索引"
"今天完成创建订单事务"
"今天使用 Apifox 测试创建订单"
"今天处理重复预约时的异常"
```

一个任务应该满足：

```text
范围小
可验证
能独立提交 Git
能写进 progress.md
能解释为什么这样做
```

## 4. 每实现一个功能，我都要回答五个问题

无论是登录、地址、订单还是评价，做完后必须自己写出以下答案。

### 问题一：入口在哪里？

```text
接口路径是什么？
HTTP 方法是什么？
前端从哪个页面调用？
Controller 的哪个方法接收？
```

### 问题二：输入是什么？

```text
请求参数有哪些？
哪些字段必填？
哪些字段需要格式校验？
参数对象 DTO 是什么？
```

### 问题三：业务校验是什么？

例如创建订单时：

```text
- 用户是否登录？
- 阿姨是否存在？
- 阿姨是否可接单？
- 地址是否属于当前用户？
- 日期是否为未来日期？
- 时段是否合法？
- 该阿姨该时段是否已被预约？
```

### 问题四：数据如何变化？

```text
操作了哪些表？
新增、更新、删除了什么？
是否需要事务？
失败后应回滚什么？
```

### 问题五：异常情况是什么？

```text
- 参数错误；
- 未登录；
- 无权限；
- 数据不存在；
- 档期冲突；
- 订单状态不允许操作；
- 数据库唯一索引冲突；
- 网络或部署错误。
```

如果我不能回答这五个问题，说明我只是"让 AI 写了代码"，还没有真正理解。

## 5. AI 的正确角色

AI 在本项目中可以充当：

| 角色 | AI 可以做什么 |
|---|---|
| 技术讲师 | 解释概念、源码、配置和报错 |
| 结对编程伙伴 | 协助实现一个小功能 |
| 代码审查者 | 检查安全、事务、空指针、命名和异常处理 |
| 测试助手 | 帮我设计接口测试、边界测试、并发测试 |
| 文档助手 | 帮我整理 README、接口文档、复盘记录 |
| 面试官 | 针对项目细节追问我 |
| 架构讨论伙伴 | 给出方案对比、优缺点和演进方向 |

AI 不应该成为：

```text
需求决定者
项目唯一设计者
我不理解代码的代写工具
我不看日志时的"报错翻译器"
```

## 6. 推荐的学习式提问模板

### 6.1 学概念

```text
请用息悦生活项目中的实际场景解释"数据库事务"。
不要只给定义，要说明：
1. 不使用事务会出现什么问题；
2. 创建订单时哪几步必须放进事务；
3. 失败后数据库会是什么状态；
4. 如何用代码和 SQL 验证事务是否生效。
```

### 6.2 做功能前

```text
我要实现"用户创建订单"，请先不要生成代码。

请帮我输出：
1. 该功能的完整业务流程；
2. 需要的 DTO、实体、表字段；
3. 需要校验的规则；
4. 事务边界；
5. 并发冲突点；
6. 建议的接口设计；
7. Apifox 测试案例。

等我确认设计后，再分步骤生成代码。
```

### 6.3 看代码

```text
这是我自己写的 OrderService 创建订单代码。

请不要直接重写。
请按以下维度审查：
1. 是否有越权风险；
2. 是否有事务问题；
3. 是否有并发重复预约问题；
4. 是否有空指针风险；
5. 异常处理是否合理；
6. 命名是否清晰；
7. 哪些地方我应该自己修改。
```

### 6.4 处理报错

```text
我遇到以下问题：

目标：
……

实际现象：
……

完整错误日志：
……

相关代码：
……

我已经尝试：
……

请先解释错误发生在哪一层：
依赖、配置、Spring 容器、数据库、网络、权限还是业务逻辑？
然后给我最小修改方案，并解释原因。
```

### 6.5 面试复盘

```text
请作为 Java 后端面试官，基于息悦生活项目追问我。

重点追问：
- JWT；
- Spring Security；
- 事务；
- 数据库唯一索引；
- 并发预约；
- 订单状态机；
- Redis；
- Docker Compose；
- Nginx；
- 权限与水平越权。

每次只问一个问题，等我回答后再点评和继续追问。
```

## 7. 遇到问题时，我的排查顺序

不要一报错就立刻让 AI 重写代码。

推荐顺序：

```text
1. 阅读完整错误信息；
2. 找到第一个真正的 Caused by；
3. 确认报错类、报错行和请求参数；
4. 查看后端控制台日志；
5. 查看浏览器 Network 请求；
6. 查看数据库实际数据；
7. 查看 Docker 容器日志；
8. 查官方文档、GitHub Issue 或版本说明；
9. 自己提出一个可能原因；
10. 再让 AI 协助分析。
```

常用排查命令和方式：

```bash
# 查看容器状态
docker compose ps

# 查看某个服务日志
docker compose logs -f backend

# 查看 MySQL 容器日志
docker compose logs -f mysql

# 查看后端接口是否可访问
curl http://localhost:8080/actuator/health

# 进入 MySQL 容器
docker exec -it <mysql-container-name> mysql -uroot -p
```

## 8. 不懂的代码不能直接合并

当 AI 给出代码后，我至少应该检查：

```text
- 这个类负责什么？
- 它为什么需要这个注解？
- 这个方法由谁调用？
- 请求从哪里进来？
- 返回值最终给谁？
- 这个 SQL 会修改什么数据？
- 事务在哪里开启？
- 如果异常发生，数据会怎样？
- 这个配置是否与当前版本兼容？
```

如果我无法回答，就先让 AI 逐行解释，或者自己调试后再继续。

---

## 9. 每完成一个模块必须留下证据

每个阶段完成后，至少保留：

- Git 提交记录；
- 关键接口截图或 Apifox 测试记录；
- 页面截图；
- 数据库结果截图或 SQL；
- `progress.md` 更新；
- 遇到的问题和解决过程；
- 自己写的一段复盘。

最终这些内容会成为：

```text
项目 README 的素材；
简历项目亮点的依据；
面试中可讲的真实经历；
以后复习 Spring Boot 项目的笔记。
```

---

## 10. 我对自己的要求

```text
不追求一天写很多代码；
追求每周真正理解几个核心问题。

不追求项目看起来像大厂系统；
追求项目中的每项技术都能讲清楚。

不把 AI 当作黑箱；
把 AI 当作讲师、审查者和协作伙伴。

不害怕报错；
把每一个报错都当成一次积累实战经验的机会。
```

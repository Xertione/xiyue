### 确认的 MVP 服务时间规则

```text
服务类型：日常保洁

用户选择服务日期；
用户选择开始时间；
用户选择服务时长；
最低服务时长为 1 小时；
服务时长按整小时计算，不允许用户输入任意分钟时间范围。

示例：
2026-08-01
10:00 开始
服务 3 小时

系统内部对应占用小时块：
10:00 - 11:00
11:00 - 12:00
12:00 - 13:00
```

> 阿姨在已被占用的具体日期和小时块不可再次接单；其他不冲突时间段仍可继续抢单。

### 确认的订单状态

| 状态值 | 名称 | 说明 |
|---:|---|---|
| 0 | 待支付 | 用户创建订单成功，尚未支付 |
| 1 | 待抢单 | 模拟支付成功，等待阿姨抢单或管理员指派 |
| 2 | 待服务 | 阿姨已抢单或被指派，等待阿姨开始服务 |
| 3 | 服务中 | 阿姨开始服务 |
| 4 | 待确认 | 阿姨提交服务完成，等待用户确认 |
| 5 | 待评价 | 用户确认服务完成，等待用户评价 |
| 6 | 已完成 | 用户已提交评价，订单闭环 |
| 7 | 已取消 | 用户主动取消或系统取消 |
| 8 | 投诉中 | 用户提交投诉，管理员处理中 |

### 完整主流程

```text
用户创建订单
→ 待支付
→ 模拟支付
→ 待抢单
→ 阿姨抢单 / 管理员指派阿姨
→ 待服务
→ 阿姨开始服务
→ 服务中
→ 阿姨提交服务完成和演示图片
→ 待确认
→ 用户确认服务完成
→ 待评价
→ 用户提交评价
→ 已完成
```

## 6. 并发预约、抢单与档期确认

| 编号 | 待确认问题 | 当前倾向 | 状态 | 最终决策 |
|---|---|---|---|---|
| C-01 | 是否要求同一阿姨同一日期同一时段只能被预约一次？ | 是 | **已确认** | 同一阿姨在同一日期、同一小时块只能接一笔有效订单。不能将阿姨整体状态直接改为"不可预约"，应只锁定该阿姨具体的日期和小时块，其他不冲突时段仍可接单。 |
| C-02 | 是否使用数据库唯一索引作为最终防线？ | 是 | **已确认** | 使用数据库唯一索引作为档期占用的最终保障。档期占用表按小时块设计，核心字段：aunt_id、service_date、hour_slot、order_id。唯一索引：`UNIQUE KEY uk_aunt_date_hour (aunt_id, service_date, hour_slot)`。 |
| C-03 | 是否在订单取消后释放阿姨档期？ | 是 | **已确认** | 只有阿姨抢单成功或管理员手动指派阿姨成功时，才锁定该阿姨对应的日期和小时块。必须在同一个数据库事务中完成：校验订单当前状态 → 校验阿姨状态和档期 → 插入档期占用 → 更新订单 aunt_id → 更新订单状态为待服务。任一步失败都必须回滚。 |
| C-04 | 待支付订单是否占用阿姨档期？ | 建议占用 | **已确认** | 待支付订单和待抢单订单均不占用任何阿姨档期。档期在阿姨抢单成功或管理员指派时才锁定。 |
| C-05 | 是否第一版就使用 Redis 分布式锁？ | 否 | **已确认** | 第一版不使用 Redis 分布式锁作为抢单和档期冲突的核心方案。最终保障采用：数据库事务 + 档期小时块占用表 + 联合唯一索引 + 订单状态条件更新。 |
| C-06 | 是否需要做并发下单测试？ | 是 | **已确认** | 需要设计并执行并发抢单测试。测试目标：多个阿姨同时抢同一订单时，最终只能有一个阿姨抢单成功；同一阿姨尝试抢时间冲突订单时，最终只能成功一次；失败请求应返回明确业务提示，不产生脏数据。 |

### 确认的并发控制原则

```text
第一版：
- 档期占用表按小时块设计（aunt_id, service_date, hour_slot）
- 使用数据库唯一索引作为最终保障
- 阿姨抢单和档期占用必须在同一个数据库事务中完成
- 待支付 / 待抢单订单不占用阿姨档期
- 订单取消后释放对应的档期占用记录
- 不使用 Redis 分布式锁作为核心方案
```

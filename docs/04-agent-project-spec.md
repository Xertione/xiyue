## 1. 项目目标

开发一个名为"息悦生活"的家政预约平台 MVP，用于学习、作品展示和简历项目。

项目必须具备完整的服务闭环：

```text
用户注册/登录
→ 浏览阿姨 / 抢单大厅
→ 创建订单（选择日期、小时段、填写地址）
→ 模拟支付
→ 订单进入待抢单
→ 阿姨抢单 / 管理员指派
→ 阿姨开始服务
→ 阿姨提交服务完成
→ 用户确认服务完成
→ 用户评价（或投诉）
→ 订单完成
```

项目不是正式商业平台，不需要实现真实支付、真实短信、地图定位、独立 App、微服务等。

## 3. 开发原则

1. 不要一次性生成完整项目；
2. 每次只实现一个明确、可验证的功能模块；
3. 开始编码前，先说明本次任务涉及的文件、表、接口和测试方式；
4. 优先解释设计思路，再给代码；
5. 不确定需求时必须提问，不允许擅自扩展功能；
6. 不引入未批准的中间件、框架或复杂技术；
7. 每完成一个模块，更新 `docs/progress.md`；
8. 每解决一个问题，更新 `docs/troubleshooting.md`；
9. 每做一个重要取舍，更新 `docs/decision-log.md`；
10. 生成代码时优先保证可读性、可调试性和学习价值，不追求过度抽象。

## 4. 角色与权限

### 普通用户

- 注册（手机号 + BCrypt 密码 + 固定验证码 `123456`）；
- 手机号 + 密码登录；
- 手机号 + 固定验证码 `123456` 登录；
- 找回密码（手机号 + 验证码 + 新密码）；
- 浏览可接单阿姨列表和详情；
- 按星级、价格、技能标签筛选阿姨；
- 发布订单（选择日期、整点开始时间、服务时长，填写联系人、电话、地址）；
- 支付自己的待支付订单；
- 取消自己的待支付、待抢单或待服务订单；服务中和待确认订单禁止直接取消；
- 查看自己的订单列表和详情；
- 确认阿姨提交的服务完成；
- 对自己的待评价订单提交一次评分+文字评价；
- 对自己的待评价订单提交一次投诉（文字原因，订单变为投诉中）。

### 阿姨

- 自行注册（手机号 + BCrypt 密码 + 固定验证码，后端仅允许 `AUNT` 角色）；
- 手机号 + 密码登录；
- 手机号 + 固定验证码登录；
- 浏览抢单大厅中的待抢单订单；
- 抢单（同一订单只能被一位阿姨抢到）；
- 查看自己的订单列表和详情；
- 对待服务订单执行"开始服务"；
- 对服务中订单执行"提交服务完成"并上传演示图片 URL；
- 设置个人接单状态（可抢单/休息）；
- 查看/编辑个人资料（姓名、价格、年龄、入行年限、技能标签、简介，见 ADR-025）；
- 设置具体日期和小时块不可接单（MVP 裁剪，见 ADR-022，当前仅实现整体接单状态切换）。

### 管理员

- 使用预置账号密码登录管理后台；
- 查看和编辑已自行注册的阿姨资料；
- 上架、下架、禁用或逻辑删除阿姨；存在进行中订单（待服务/服务中/待确认）时拒绝删除，历史数据保留（ADR-015）；
- 查看全量订单，按状态筛选；
- 在后台手动指派阿姨到待抢单订单（兜底机制）；
- 查看评价；
- 查看投诉并手动处理。

## 5. 业务规则

### 5.1 服务规则

```text
服务品类：仅"日常保洁"
时间选择：用户选择日期 + 整点开始时间 + 整点服务时长（最低 1 小时）
档期单位：小时块（如 09:00-10:00、10:00-11:00）
```

### 5.2 价格规则

```text
抢单订单：用户发布时订单已确定金额，阿姨按订单金额结算；
指定阿姨订单（辅助）：按阿姨当前标价计算（MVP 裁剪，见 ADR-023，当前仅支持抢单模式）。
订单必须保存价格快照。
```

### 5.3 预约与抢单规则

```text
- 用户只能预约未来日期；
- 待支付订单和待抢单订单不占用任何阿姨档期；
- 阿姨抢单或管理员指派阿姨成功时，才锁定对应日期和小时块；
- 一个订单只能被一位阿姨抢到；
- 同一阿姨同一日期同一小时块只能接一笔订单；
- 订单取消后释放对应的档期占用记录；
- 不支持改期；
- 不支持多个阿姨共同服务一个订单。
```

### 5.4 状态机

```text
待支付(0)
  ├─ 模拟支付成功 ──> 待抢单(1)
  └─ 用户取消 ────> 已取消(7)

待抢单(1)
  ├─ 阿姨抢单/指派 ──> 待服务(2)
  └─ 用户取消 ────> 已取消(7)

待服务(2)
  ├─ 阿姨开始服务 ──> 服务中(3)
  └─ 用户取消 ────> 已取消(7)

服务中(3)
  ├─ 阿姨提交完成 ──> 待确认(4)
  └─ 禁止直接取消，MVP 由管理员线下介入

待确认(4)
  ├─ 用户确认完成 ──> 待评价(5)
  └─ 禁止直接取消，MVP 由管理员线下介入

待评价(5)
  ├─ 用户评价 ────> 已完成(6)
  └─ 用户投诉 ────> 投诉中(8)

投诉中(8)
  └─ 管理员处理 ──> 已完成(6)
```

禁止非法状态流转。

## 6. 数据库要求

核心表：

```text
sys_user         用户/阿姨/管理员账号
aunt             阿姨资料（个人信息 + 运营状态）
service_order    预约订单（含地址快照、价格快照）
aunt_booking_slot 阿姨小时块档期占用记录
review           用户评价
complaint        用户投诉（简化版）
```

要求：

1. `sys_user.phone` 必须唯一；
2. `sys_user.role` 区分 USER / AUNT / ADMIN；
3. 所有用户的密码必须使用 BCrypt 加密存储；
4. `service_order.order_no` 必须唯一；
5. `aunt.user_id` 必须唯一并关联自行注册的阿姨账号；
6. `aunt_booking_slot` 必须存在唯一索引：

```sql
UNIQUE KEY uk_aunt_date_hour (aunt_id, service_date, hour_slot)
```

7. `review` 必须存在唯一索引：

```sql
UNIQUE KEY uk_review_order (order_id)
```

8. `complaint.order_id` 必须唯一；
9. 金额字段使用 `DECIMAL(10,2)`；
10. Java 金额类型使用 `BigDecimal`，禁止使用 `double`；
11. 订单必须保存地址、阿姨和价格快照；
12. 已支付订单取消时记录模拟退款状态、流水号和时间；
13. 阿姨存在两类独立状态字段：管理状态（可用/下架/禁用）和接单状态（可抢单/休息），禁止混合；
14. 所有表使用 `utf8mb4`；
15. 关键表必须包含创建时间和更新时间字段。

## 7. 核心技术要求

### 7.1 认证

- 注册接口：手机号 + 密码（BCrypt） + 固定验证码 + 角色；公开角色只允许 `USER` 或 `AUNT`；
- 登录接口：手机号 + 密码 或 手机号 + 固定验证码；
- 找回密码接口：手机号 + 固定验证码 + 新密码；
- 固定验证码缓存在 Redis，Key：`sms:code:{phone}`，TTL 5 分钟；
- 固定验证码值为 `123456`。
- 管理员账号由应用启动时初始化（AdminAccountInitializer，ADR-013），禁止公开注册；
- 后端必须对白名单角色进行校验，禁止客户端构造 `ADMIN` 或其他角色；
- JWT 中的角色必须从数据库读取，不能直接采用请求体中的角色。

### 7.2 安全

- 使用 Spring Security + JWT Filter；
- Token 从 `Authorization: Bearer <token>` 获取；
- 未登录请求受保护接口必须返回未认证错误；
- 不同角色访问不属于自己的接口必须被拒绝；
- 用户访问订单时必须校验资源归属；
- 阿姨只能操作自己接的订单。

### 7.3 阿姨抢单事务

阿姨抢单或管理员指派阿姨，必须在同一个数据库事务中完成：

```text
校验订单当前状态为待抢单
→ 校验阿姨管理状态为可用
→ 校验阿姨该日期和小时块未被占用
→ 使用 `WHERE status = 待抢单 AND aunt_id IS NULL` 条件更新订单归属和状态
→ 校验订单更新行数必须为 1
→ 插入全部档期占用记录
```

任一步失败都必须回滚。

如果档期唯一索引冲突，应返回明确业务错误：

```text
该阿姨在该时段已被预约，请选择其他阿姨。
```

第一版不使用 Redis 分布式锁作为抢单和档期冲突的核心方案。
档期唯一索引只防止同一阿姨时段冲突；同一订单只能被抢一次必须依靠订单条件更新。订单更新与档期插入必须在同一事务中完成，任一步失败整体回滚。

### 7.4 支付

第一版使用 `MockPaymentService`：

- 仅允许待支付订单发起模拟支付；
- 模拟支付成功后生成模拟支付流水号，记录支付时间和支付方式；
- 更新订单状态为待抢单；
- 不实现真实微信支付；
- 保留支付服务抽象接口，但不要虚构已实现微信支付能力。

### 7.5 定时任务

第一版不设置强制自动取消规则，也不引入 Spring Task。

后续如果需要，可增加：

- 长时间无人抢单的提示或通知（第一版不做）。

### 7.6 取消与模拟退款

- 仅待支付、待抢单、待服务订单允许用户取消；
- 服务中和待确认订单禁止直接取消，MVP 由管理员线下介入；系统投诉仅在待评价状态开放；
- 取消使用状态条件更新，防止与抢单、开始服务并发成功；
- 已支付订单记录模拟退款状态、流水号和时间；
- 已锁定档期的订单取消时，在同一事务中释放档期。

### 7.7 订单快照

订单创建时保存联系人、电话、地址、订单金额快照。阿姨抢单成功时将阿姨姓名、头像写入订单快照。

### 7.8 阿姨状态分离

阿姨管理状态字段（admin 控制）：

```text
AVAILABLE（可用） / OFF_SHELF（下架） / DISABLED（禁用）
```

阿姨接单状态字段（阿姨/系统控制）：

```text
AVAILABLE（可抢单） / RESTING（休息）
```

档期占用由 `aunt_booking_slot` 表管理，不与以上字段混用。

### 7.9 投诉约束

- 仅待评价订单允许提交投诉；
- 同一订单最多一条投诉；
- 投诉处理使用状态条件更新，禁止重复处理；
- 管理员处理后订单进入已完成，不再允许评价，也不更新阿姨评分。

### 7.10 密钥与敏感配置

- JWT 签名密钥、MySQL 密码、Redis 密码、管理员初始密码不得写入源码、文档示例、日志或 Git；
- `.env`、`application-local.yml`、私钥文件、证书私钥和数据库备份必须被 Git 忽略；
- 仓库只保留不含真实值的 `.env.example`，敏感值通过环境变量注入；
- JWT 密钥必须使用密码学安全随机值，至少 32 字节，不得使用项目名、手机号或默认短字符串；
- 日志不得输出完整 JWT、密码、验证码、数据库连接串密码或环境变量值；
- 密钥疑似泄露时必须立即轮换，旧 Token 失效，并将事件记录到 `docs/troubleshooting.md`。
